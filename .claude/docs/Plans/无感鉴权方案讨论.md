# HTCMC PCHSystem 无感鉴权方案讨论

> **状态**：讨论稿，**暂不实现**。记录离线服防冒名鉴权的方案选型与结论。
> **日期**：2026-07-01
> **配套**：第一阶段 MVP 计划见 [`MVP-第一阶段计划.md`](./MVP-第一阶段计划.md)

---

## 1. 问题背景

离线模式 `UUID = MD5("OfflinePlayer:"+name)` 可推导，服务器不向 Mojang 验证身份——**任何客户端可用任意名字连入**。冒名攻击路径：

1. 攻击者获知某 admin/玩家名 → 用该名字连入服务器
2. 进服执行 `!!login` → MCDR 用 `uuid_api_remake` 拿到正主 UUID → 后端签发 token URL
3. 攻击者点击 URL → **盗用正主 Web 身份**

[AutoModpack](https://github.com/Skidamek/AutoModpack) 是 modpack 分发工具，其「authorization」针对的是 mod 文件下载通道，**不验证玩家身份**，对冒名零防护（[Issue #335](https://github.com/Skidamek/AutoModpack/issues/335)）。这与「装载相关模组时被攻击过」的经历吻合。

---

## 2. 威胁模型与约束

- **威胁来源**：**外部批量扫描**为主（非内部针对性攻击）→ 门槛够挡扫描即可。
- **整合包内部发行、不对外** → 客户端环境可控，可内嵌凭证。
- **玩家构成**：正版 + 离线**混合** → 不能开 `online-mode=true`，纯 Mojang 验证不可行。
- **密钥不进 AutoModpack**（防 modpack 同步外泄）。
- **诉求**：零手动批准 + 无感 + 不自研复杂 challenge-response 握手（之前尝试写不好）。

---

## 3. 纯 MCDR 不可行（已确认）

MCDR 基于 RCON/控制台 IO，**拿不到连接握手 / configuration phase 的网络包**，只能进服后用命令交互 = 变相密码（需冻结期），非无感。无感鉴权必须发生在**玩家进世界之前**，只能由 **Fabric mod**（configuration phase 挑战-响应）或**服务端 Java agent**（authlib-injector）完成。

> 用户「MCDR 或 Fabric 实现」的诉求里，**MCDR 路线排除**。

---

## 4. 方案 A：McGatekeeper（现成，备用方案）

[TheGameratorT/McGatekeeper](https://github.com/TheGameratorT/McGatekeeper) —— Fabric mod，Ed25519 非对称握手。

**机制**：
- 客户端首次连服自动生成 Ed25519 密钥对（每服务器独立，存 OS 数据目录，**不进 modpack**，导出带不走）。
- 认证在 **configuration phase** 完成（玩家进世界前）：服务端发 nonce + 服务端签名 → 客户端验签（防中继）→ 客户端用私钥签名 → 服务端验批准列表 → 放行。
- 首次连服需 admin `/gate allow <player> <label>` 批准设备；之后全自动。
- `/gate reset <player> <label>` 吊销指定设备。

**优点**：批准后真无感；每设备一钥可单独吊销；安全性最强；零自研。
**痛点**：每个新玩家**首次**连服需 admin 手动批准一次（社区服新玩家不多时可接受）。

---

## 5. 方案 B：皮肤站（已放弃）

authlib-injector + [Blessing Skin Server](https://github.com/bs-community/blessing-skin-server)。功能全（UUID 由站下发不可推导 + 皮肤 + 正版混合 B1 统一接管），但**运维 PHP/MySQL 太重**，用户已否决。

> 若未来愿意运维皮肤站，可重新评估。正版混合方案：B1（皮肤站装正版验证插件，正版玩家 OAuth 一次建号，UUID 用 Mojang 的；离线玩家注册，UUID 由站下发）。

---

## 6. 方案 C：PSK 预共享密钥 + config 内嵌（主方案）⭐

### 机制
- 整合包内嵌一个客户端 mod，其 config 含**共享密钥 PSK**（admin 生成，不通过 AutoModpack 同步，直接打进整合包发行）。
- 玩家装整合包即持有 PSK；连服时客户端 mod 在 configuration phase 向服务端证明持有 PSK：
  - 服务端发随机 nonce
  - 客户端回 `HMAC-SHA256(PSK, nonce)` + 玩家名
  - 服务端用存储的 PSK 重算 HMAC 比对 → 通过放行 / 拒绝踢出
- **零手动批准**。

### 安全性（对应威胁模型）
- ✅ 外部扫描者拿不到整合包 → 无 PSK → 进不来。
- ⚠️ **共享密钥**：任一内部玩家外流整合包 = 全服暴露；无法区分个体 / 单独吊销。
- ⚠️ **密钥轮换**：下次整合包更新换 PSK 时，**旧版整合包玩家必须更新才能进**。
- 结论：对「外部扫描」威胁**够用且合适**；内部威胁不在此方案防护范围。

### 实现路径（卡点：握手自研）
- **客户端 mod**：读 config 的 PSK → configuration phase 监听服务端 nonce → 回 HMAC + 玩家名。
- **服务端 mod**：configuration phase 发 nonce → 验 HMAC → 放行 / 踢出。
- 用 Fabric Networking API（`ServerConfigurationNetworkingEvents` / `ConfigurationPayload`）。逻辑比 McGatekeeper 的 Ed25519 简单（对称 HMAC vs 非对称签名），但 **configuration phase 握手时序**是用户「写不好」的部分——可由 AI 辅助实现。
- **现成性**：已知 Fabric 鉴权 mod（EasyAuth / McGatekeeper / Login System / Authenticate / Offline Authenticator）**均非 PSK-config 模式**，大概率自研或魔改。**执行前需检索 Modrinth / CurseForge 确认**。

---

## 7. 方案 D：McGatekeeper + MCDR 自动批准（可选折中）

- 用 McGatekeeper 保留非对称安全；MCDR 监听「等待批准」状态，对预白名单内玩家自动执行 `/gate allow`。
- 零手动批准 + 每设备一钥可吊销。
- **待验证**：McGatekeeper 是否暴露「等待批准」事件、能否被 MCDR / 外部命令触发批准（需读源码或测试）。

---

## 8. 对比与推荐

| 方案 | 定位 | 手动批准 | 自研量 | 安全性 |
|---|---|---|---|---|
| **C PSK config** ⭐ | **主方案** | 零 | 中（握手 mod） | 够挡扫描 |
| A McGatekeeper | **备用方案（用户指定）** | 每新玩家 1 次 | 零（现成） | 最强 |
| D McGatekeeper+MCDR 自动 | 可选折中 | 零 | 小（脚本） | 最强（需验证可外部触发） |

- **主方案 C（PSK）**：契合「零手动 + 内嵌密钥」诉求，威胁模型下安全性足够，握手（HMAC nonce）可 AI 辅助实现。
- **备用方案 A（McGatekeeper）**（用户确认保留）：若 PSK 自研受阻，或安全需求升级（需每设备可单独吊销 / 防内部整合包外流），回退 McGatekeeper——现成、最安全，代价是新玩家首次手动批准一次。
- 方案 D 作为可选折中保留。

---

## 9. 待决策 / 待调研

1. 是否接受 PSK 共享密钥的「外流即全暴露」风险？（威胁模型是外部扫描，应可接受）
2. PSK 轮换策略：每次整合包大版本换？还是不轮换？
3. **执行前检索** Modrinth / CurseForge 是否有现成 PSK-config 鉴权 mod（避免重复造轮子）。
4. 若选方案 D：验证 McGatekeeper 能否被 MCDR 外部触发 `/gate allow`。
5. 单一 PSK vs 多 PSK（按批次 / 按玩家组分钥，提升隔离）。

---

## 10. 对第一阶段影响

- 选 C / D / A 任一：UUID 仍 OfflinePlayer（`uuid_api_remake` 获取），但冒名者无 PSK / 密钥进不来 → 进服身份可信。
- `!!login` token 流程保留（进服身份可信后，token 登录 Web）。
- 鉴权定稿前，第一阶段身份流程的**进服环节**暂停细化；**表格部分**（与鉴权无关）可先行。
